# TIFF 取り扱いメモ

## 結論

このビューアでは、TIFF をブラウザで直接デコードして表示する方式は採用しません。TIFF はバックエンドでページごとに PNG 化し、フロントエンドはその画像を表示する方式を正とします。

## 発生した問題

### 1. `UTIF.js` で strip ノイズが発生

- 一部の TIFF で、横方向にノイズが走る
- 画像全体は表示されるが、strip 境界ごとに `じゃみじゃみ` した崩れが入る
- 実データでは日本語の帳票 TIFF で再現した

### 2. `geotiff.js` でも `Invalid array length`

- `UTIF.js` の代替として `geotiff.js` を試した
- しかし LZW 圧縮 TIFF で `Invalid array length` が発生した
- 直接デコードを続けると、TIFF の種類ごとにフロントの実装負荷が増える

### 3. クライアントデコードは読み込みも重い

- TIFF をブラウザ側でデコードすると、PDF/JPEG に比べて初回表示が明らかに遅い
- 複数ページ TIFF や圧縮 TIFF ほど負荷が大きい
- フロントの責務としては重すぎる

## 採用した方針

- バックエンドで TIFF を開く
- ページ数をバックエンドで取得する
- 必要なページを `viewer2d/sessions/{id}/pages/{page}/image` から PNG として返す
- フロントは PNG を通常画像として描画する

## この方針のメリット

- TIFF の圧縮方式差異をフロントで抱えなくてよい
- ノイズや decode 例外が出にくい
- フロントの初回読み込みが軽くなる
- 2D 描画系を PDF/JPEG と同じ「画像表示」寄りに寄せられる

## 今後の扱い

- TIFF 直デコードを再導入する場合は、実データでの再検証を前提にする
- TIFF 表示不具合が再発したら、まずバックエンドのページ画像化経路を疑う
- 新しい TIFF サンプルを受け取った場合は、圧縮方式とページ数を記録して検証する
- `pageCount` と `pageImageUrls` を返す API 契約を変える場合は、`docs/viewer-specification.md` と `docs/integration-manual.md` も同期する
