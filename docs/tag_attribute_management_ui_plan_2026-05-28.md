# タグ・属性管理UI計画

- 作成日: 2026-05-28
- 目的: 図面管理におけるタグ・属性の登録、確認、補正、再抽出の UI を実装可能な粒度で整理する。
- 前提:
  - 本体ナレッジシステムのソースは無い。
  - `2D_3D_CAD_VIEWR` の図面詳細 UI を参考に、後で本体へ組み込みやすい形で設計する。

## 1. この UI の位置づけ

- viewer は `表示` が主目的
- この UI は `管理` が主目的
- したがって、以下を重視する。
  - 属性の一覧性
  - 根拠確認
  - 手動補正
  - 再抽出
  - ステータス管理

## 2. 主要画面

### 2.1 図面一覧画面

目的:

- 登録済み図面の検索
- 抽出状態確認
- タグ・属性ベース絞り込み

主要要素:

- 検索バー
- フィルタ
  - 客先
  - 案件
  - 装置カテゴリ
  - 文書種別
  - 形式
  - 抽出状態
- 一覧テーブル
  - 図面番号
  - 図面名
  - 客先
  - 案件
  - 装置カテゴリ
  - 主要タグ
  - 抽出状態
  - 更新日時

### 2.2 図面詳細管理画面

目的:

- 自動抽出結果の確認
- 手動補正
- viewer / RAG に渡る前の確定

主要セクション:

- 基本情報
- 属性情報
- タグ一覧
- 抽出根拠
- 手動補正
- 再抽出履歴

### 2.3 再抽出ジョブ画面

目的:

- ジョブ状態確認
- 失敗時の再実行
- エラーログ確認

## 3. 図面詳細管理画面の構成

## 3.1 上部ヘッダー

- 図面番号
- 図面名
- 版数
- 形式
- 抽出状態
- 最終更新日時

右上アクション:

- `再抽出`
- `補正保存`
- `RAG更新`

## 3.2 基本情報カード

viewer の `DrawingOverviewPanel` を参考に、以下を表示する。

- 図面番号
- 図面名
- 図面種別
- 客先名
- 案件名
- 装置名
- 装置カテゴリ
- 用紙サイズ
- ステータス

## 3.3 属性情報セクション

グループ表示を推奨する。

- 共通属性
- 3D 属性
- 2D 属性
- 業務補助属性

各行に持たせたいもの:

- 項目名
- 値
- 抽出元
- 信頼度
- 手動補正有無

## 3.4 タグセクション

表示形式:

- タグチップ一覧

操作:

- タグ削除
- タグ追加
- 自動タグ/手動タグの表示分離

## 3.5 抽出根拠セクション

目的:

- 「なぜその属性/タグが付いたか」を追えるようにする

表示内容:

- top part 任意情報断片
- part 名一覧
- 2D 抽出文字列断片
- 寸法/公差断片
- OCR 使用有無

## 3.6 手動補正セクション

操作対象:

- 属性値の上書き
- タグの追加/削除
- 補正理由

保持項目:

- 更新者
- 更新日時
- 補正理由

## 3.7 再抽出履歴セクション

- 抽出時刻
- extractor version
- elapsed_ms
- status
- warning 件数

## 4. 一覧画面のフィルタ仕様

### 必須フィルタ

- 客先名
- 案件名
- 装置カテゴリ
- 図面種別
- source format
- 抽出状態

### 追加フィルタ候補

- 主要タグ
- OCR 使用有無
- 手動補正あり/なし

## 5. 画面遷移

1. 図面一覧
2. 図面詳細管理
3. 必要に応じて viewer 詳細
4. 必要に応じて再抽出ジョブ詳細

## 6. `2D_3D_CAD_VIEWR` を参考にする点

### 参考にする

- 基本情報カードの構成
- 属性情報と備考の見せ方
- 図面詳細画面の密度感

### そのまま持ち込まない

- mock の改訂履歴や関連情報
- viewer の操作ボタン群
- 表示専用前提の軽量 metadata 契約

## 7. Django 実装への落とし方

### テンプレート/SPA どちらでも共通の責務

- 一覧データ取得 API
- 詳細データ取得 API
- 補正保存 API
- 再抽出 API
- ジョブ状態 API

### 2026-05-28 実装反映

- 今回は first step として Django template で以下を実装済み
  - `/drawing-metadata/`
  - `/drawing-metadata/{drawing_id}/`
  - `/drawing-metadata/jobs/{job_id}/`
- 役割:
  - 一覧
  - 詳細確認
  - 手動補正 JSON 保存
  - 再抽出起票
  - ジョブエラー確認
- 本体へ移植する段階で React / 既存画面へ載せ替えてもよいが、まずは backend 契約の確認を優先する

### 最小構成

- 一覧
- 詳細
- 補正保存
- 再抽出

### 後回しにしてよいもの

- 高度な監査検索
- 一括編集
- タグ辞書メンテ UI

## 8. 実装優先順位

1. 一覧画面
2. 詳細画面の基本情報/属性情報
3. タグセクション
4. 補正保存
5. 再抽出ボタンとジョブ状態
6. 抽出根拠表示

## 9. 受入条件

- 図面一覧で客先・案件・装置カテゴリで絞り込める
- 詳細画面で属性・タグ・抽出根拠が見える
- 手動補正を保存できる
- 再抽出を起動できる
- viewer と同じ図面 ID で連携できる

## 10. 結論

- 本体ソースが無い現状では、viewer を参考にしつつ、管理用 UI は別物として起こすのが妥当である。
- この UI の本質は「閲覧」ではなく「抽出結果の確認と補正」である。
- 後で本体へ組み込む前提なら、一覧/詳細/補正/再抽出の最小セットから作るのが最も安全である。
