## 2. 事実として確認できたこと

### 2.1 `knowledge_system` の位置づけ

- `knowledge_system` はナレッジシステム本体コードのリポジトリではなく、RAG 精度検証、報告書作成、補助スクリプト整理が中心である。
- `scripts` 配下には Excel/報告書の生成・追記スクリプトがあり、ナレッジシステム本体の API や DB 実装は含まれていない。
- 既存整理資料では、RAG の主要課題として以下が繰り返し挙がっている。
  - 案件混線
  - 客先名/案件名/カテゴリ名の切り分け不足
  - BOM/部品情報から案件名へ上がれない
  - 規格表や Word/Excel 仕様書の読取り精度不安定

### 2.2 `図面管理` はタグ・属性の正本候補として既に想定されている

- 既存整理資料では、図面登録画面に対応形式として `.png`, `.jpg`, `.jpeg`, `.tiff`, `.pdf`, `.icad`, `.step`, `.stp` が表示されていたと記録されている。
- 同じ資料には「タグは登録後に図面から自動で抽出されます。手動で追加・削除も可能です。」という文言が記録されている。
- 一方で、この表示は「予定仕様またはモック表示として扱うべき」と明記されており、本番実装済みの証拠にはなっていない。
- `http://210.165.3.139/web/drawing` への到達自体は確認できたが、静的 HTML では SPA の殻と JavaScript 資産参照しか取得できず、実際のデータ契約や項目構造は確認できていない。

### 2.3 `2D_3D_CAD_VIEWR` の viewer は属性の消費先であり、現時点で正本ではない

- `2D_3D_CAD_VIEWR` は PDM の図面詳細画面へ埋め込む専用 viewer として整理されている。
- viewer の現行 bootstrap metadata は、以下の軽量情報に留まっている。
  - `drawingNumber`
  - `drawingName`
  - `drawingType`
  - `paperSize`
  - `status`
  - `owner`
  - `designPurpose`
  - `tags`
- viewer の補助セクションである `属性情報`、`備考`、`改訂履歴`、`関連情報`、`変更履歴` は、現在 `drawingKnowledge.ts` の mock detail で補完している。
- したがって viewer は、タグ・属性の正本保存先ではなく、図面管理または関連 API から受け取った情報の表示先として扱うのが自然である。

### 2.4 `sxnet` の 3D 側では、部品階層と部品詳細がかなり深く取れそうである

- `SxWF.getInfPartTree()` は、最上位パーツからの階層情報を取得し、解説に「パーツ階層構造、パーツ詳細情報、パーツ任意情報を一括して取得できる」とある。
- `SxInfPartTree` では以下の情報が保持される。
  - `child_list`
  - `parent`
  - `inf`
  - `ex_inf`
  - `entpart`
- `SxInfPart` では以下の部品詳細が確認できる。
  - `name`
  - `comment`
  - `ref_model_name`
  - `path`
  - `is_external`
  - `is_mirror`
  - `is_read_only`
  - `is_unloaded`
- `SxWF.getInfExTopPart()` では、3D グローバル WF の最上位パーツ任意情報文字列が取得できる。
- 3D 側は、単なる形状表示用途ではなく、BOM、案件識別、設計単位識別、検索属性生成に使える構造化入力源になりうる。

### 2.5 `sxnet` の 2D 側では、OCR 以外に文字・寸法・注記・溶接情報を直接取れる見込みがある

- `SxEntSeg.getGeom()` / `getGeomList()` は、セグメント種別に応じて `SxGeom` 派生オブジェクトを返す。
- その中には少なくとも以下が含まれる。
  - `SxGeomText`
  - `SxGeomLabel`
  - `SxGeomLengthDim`
  - `SxGeomAngDim`
  - `SxGeomDiaDim`
  - `SxGeomChamDim`
  - `SxGeomWeld`
  - `SxGeomBalloon`
- `SxGeomText` では以下が確認できる。
  - `txt`
  - `text_line_num`
  - `atr_word`
  - `pnt`
  - `base_pnt`
- `SxDimValueAtr` では以下が確認できる。
  - `dist1`
  - `dist2`
  - `frontword`
  - `backword`
  - `downword`
  - `upper_tolword`
  - `down_tolword`
  - `mark2`
  - `mark3`
  - `opt_word`
  - `valword`
  - `str_num`
- 2D ICAD については、OCR に頼らずとも、文字列、注記、寸法値、公差、記号、溶接注記などを直接抽出できる可能性が高い。

### 2.6 実行環境側の前提

- `Python 3.11.9` が利用可能である。
- `clr` モジュールは検出できたため、`pythonnet` 系の実行土台はある。
- ただし、`knowledge_system` ワークスペース内には、`sxnet` や `pythonnet` を使った既存コードは見当たらない。
- 現在の `sxnet` フォルダは HTML リファレンス中心であり、PoC を進めるには実際の参照 DLL の置き場所とロード方法を明確化する必要がある。

### 2.7 形式別の扱いは既存整理資料でも分かれている

- 既存資料では、タグ自動取得の実装タスクが以下の単位に分かれている。
  - 2D 画像 (`png/jpeg/tiff`)
  - 2D PDF
  - 2D 共通のテキストからのタグ・属性化
  - 3D STEP
  - 3D ICAD
  - 共通のタグ・属性設定
- 同じ資料では、2D は「文字情報ベース」、非サーチャブル図面は OCR が課題、3D は「属性情報抽出」が主眼と整理されている。

## 4. 現行システムとのギャップ

### 4.1 `図面管理` のギャップ

- 自動抽出対象項目の正式定義がまだ見えていない。
- 抽出結果をどこへ保存するか、どこまで手動補正できるかが未確定である。
- 図面一覧・絞り込み・詳細表示・RAG 連携で、同じ属性を共通利用する前提がまだ明文化されていない。

### 4.2 viewer のギャップ

- bootstrap metadata は軽量で、図面詳細に見せたい属性群を保持していない。
- 補助セクションは mock のため、実データに差し替える別 API または bootstrap 拡張が必要である。
- viewer は表示先としては十分な受け皿があるが、保存責務は持っていない。

### 4.3 RAG のギャップ

- 既存整理資料で確認されている失敗傾向は、自然言語だけで解ける問題ではなく、事前メタデータ不足の問題が大きい。
- 特に以下は、タグ・属性整備で改善余地が大きい。
  - 客先名、案件名、装置カテゴリの事前フィルタ
  - 部品名から案件名へ上がるための構造化リンク
  - 文書種別ごとの読取り戦略切り替え
  - 負例時に無関係資料を出さない制御

## 5. 抽出速度に関する調査結果

### 5.1 事実として確認できたこと

- `sxnet` には、一括取得を前提にした API がある。
  - `SxEntSeg.getGeomList()` は「指定セグメント群のジオメトリを一括取得します」と記載されている。
  - `SxEnt.getInfList()` は「指定要素群の情報を問い合わせます」として配列取得を提供している。
  - `SxWF.getInfPartTree()` / `SxEntPart.getInfPartTree()` は、階層構造、パーツ詳細情報、パーツ任意情報を一括取得できると記載されている。
- 3D viewer 側の既存ドキュメントでは、`STEP -> STL` 変換は「重い STEP の変換時間」がボトルネックとして明記されている。
- viewer の bootstrap は「重い 3D 変換を行わない」と明記されており、重処理を避ける設計になっている。
- TIFF については、既存 viewer ドキュメントで「ブラウザ側で直接デコードすると初回表示が明らかに遅い」と整理され、バックエンドで PNG 化する方式へ寄せている。
- 既存整理資料では、2D 共通は「テキストからタグ・属性化」、3D ICAD は「情報の取得、タグ・属性化」と整理されており、少なくとも初期フェーズでは形状変換よりメタデータ抽出が中心課題である。

### 5.2 速度見立ての前提

- 以下の見立ては、実測ではなく、API の構造、既存 viewer のボトルネック整理、処理内容の重さから見た設計判断用の推定である。
- 想定条件:
  - 同一 PC または同一 LAN 上から対象ファイルへアクセスできる
  - 図面登録時の対象は 1 ファイル単位
  - 3D はメッシュ生成ではなく、部品階層・任意情報などの属性抽出が主
  - 2D は OCR 前提ではなく、ICAD ネイティブ文字・寸法抽出を優先する

### 5.3 形式別の速度見立て

| 形式/処理 | 速度見立て | 理由 | 実装上の扱い |
| --- | --- | --- | --- |
| 3D ICAD の部品階層・任意情報抽出 | 速い寄り | `getInfPartTree()` で階層・詳細・任意情報を一括取得でき、メッシュ変換や OCR を挟まないため | 登録時同期処理の候補。ただし大規模アセンブリではバックグラウンド化も視野 |
| 2D ICAD の文字・寸法・注記抽出 | 速い寄り〜中程度 | `getGeomList()` で一括取得でき、OCR を使わず文字・寸法へ直接到達できるため | 登録時同期処理の有力候補。図面サイズ・セグメント数が多い場合は計測要 |
| サーチャブル PDF のテキスト抽出 | 中程度 | OCR ほど重くないが、ページ数や表構造の復元負荷が乗るため | 小規模資料は同期可。複数ページ・表多め資料は非同期寄りが安全 |
| STEP の属性抽出または表示向け変換 | 遅い寄り | 既存 viewer でも `STEP -> STL` 変換を重い処理として分離しているため | 図面登録の同期処理に入れず、ジョブ化前提で扱うのが安全 |
| TIFF/JPEG/PNG の OCR 抽出 | 最も遅くばらつき大 | TIFF 直処理でも重いと整理されており、OCR はさらに画像解析コストが乗るため | 常にバックグラウンドジョブ前提。進捗表示とキャッシュが必要 |

### 5.4 条件付きの推定レンジ

- ここでのレンジは、PoC 前に優先度を決めるための条件付き見積もりであり、検収値ではない。

| 処理 | 条件付き推定レンジ | 備考 |
| --- | --- | --- |
| 3D ICAD メタデータ抽出 | サブ秒〜数秒 | 階層深さ、部品数、外部参照解決の有無で変動 |
| 2D ICAD 文字・寸法抽出 | サブ秒〜数秒 | セグメント数、注記量、図面サイズで変動 |
| サーチャブル PDF テキスト抽出 | 数秒前後〜十秒未満 | ページ数と表の多さで変動 |
| STEP 変換/重解析 | 数秒〜数十秒 | viewer でも重処理扱い。サイズ依存が大きい |
| OCR 抽出 | 数秒〜数十秒以上 | 解像度、ページ数、圧縮方式、言語混在で大きく変動 |

### 5.5 設計上の含意

- 初期フェーズで最も費用対効果が高いのは、`3D ICAD` と `2D ICAD` のネイティブ抽出である。
- `STEP` と `OCR` は、同期 UI に直接ぶら下げると体感待ち時間を悪化させやすい。
- したがって、図面管理の登録フローは以下で分けるのが自然である。
  - 速い処理:
    - ICAD 3D の階層/任意情報抽出
    - ICAD 2D の文字/寸法/注記抽出
  - 遅い処理:
    - STEP の重解析または変換
    - OCR
- RAG 改善を急ぐなら、まず高速に回せる ICAD ネイティブ抽出から canonical attribute を作り、OCR や STEP は第2段階に回す方が実装効果が高い。

## 6. 推測として扱うべきこと

- 実態としては、アルパイン側が ICAD ネイティブ抽出と属性正規化を担当し、創屋側が図面管理・検索・クロール・RAG 連携へ接続する分担がもっとも自然に見える。
- `図面管理`、viewer、RAG が同じ canonical attribute を共有しない限り、別案件混線や詳細表示の手修正コストは減りにくい。
- viewer の `tags` フィールドだけを増やすより、図面管理で canonical attribute を正本化し、viewer 側では read-only 参照に寄せた方が保守しやすい。
- 2D ICAD は OCR 中心ではなく、ネイティブ文字・寸法抽出中心で考えた方が精度・保守性ともに有利である。

## 10. 結論

- 今回の調査結果から見ると、タグ・属性の正本は viewer ではなく `図面管理` に置くべきである。
- 3D ICAD は `部品階層 + 部品詳細 + 任意情報`、2D ICAD は `文字 + 寸法 + 注記 + 記号` を中核データとして扱える可能性が高い。
- RAG の既存課題は、単純なプロンプト改善より、図面管理を起点とした canonical attribute 整備で改善する余地が大きい。
- 次に進むべきは実装着手ではなく、保存モデル、属性定義、viewer/RAG 連携方針を明文化した設計計画書と、そこから逆算した PoC バックログの整備である。
