# ICADタグ・属性 実装引継ぎバックログ

- 作成日: 2026-05-26
- 目的: 次会話以降で実装を始める際に、探索をやり直さず着手できるようにする。
- 前提: `図面管理` をタグ・属性の正本とし、viewer と RAG は参照・利用側に寄せる。

## 1. このバックログの使い方

- まず `B01` と `B02` の PoC で、何が安定抽出できるかを実データで確定する。
- その結果を受けて `B03` と `B04` で canonical schema と保存契約を固定する。
- viewer と RAG の接続作業は、それ以降に進める。

## 2. 次会話で最初に着手すべき項目

### 推奨開始点

- `B01: ICAD 3D 抽出 PoC`

### 理由

- 3D 側は `sxnet` 上で取得できる構造が比較的明確で、部品階層と任意情報が RAG/図面管理の両方に効く。
- 3D PoC の結果が出ると、canonical attribute の軸が決まりやすい。
- 2D PoC は 3D よりも文字の意味付けルール設計が重いため、3D の結果を踏まえて進めた方が判断しやすい。

## 7. 開発先へ確認する質問の束

1. `図面管理` をタグ・属性の正本にしてよいか。
2. 抽出結果の保存先は何か。テーブルか JSON か検索インデックス直書きか。
3. viewer の詳細情報は別 API で返す想定にしてよいか。
4. ICAD ネイティブ抽出はアルパイン担当、保存/検索接続は創屋担当でよいか。
5. OCR を正式対象に含めるか。
6. STEP は検索属性抽出も対象か、表示変換用途だけか。
7. 手動補正履歴を保持する必要があるか。

## 8. 受入条件

### Phase 1 完了条件

- 3D/2D それぞれで、実データから安定して取れる項目が明文化されている。
- 抽出結果の JSON 形が 1 つにまとまっている。

### Phase 2 完了条件

- `図面管理` で持つ canonical attribute が確定している。
- 自動抽出値と手動補正値の扱いが決まっている。

### Phase 3 完了条件

- viewer が mock detail ではなく実データを受け取る契約が定義されている。

### Phase 4 完了条件

- RAG が canonical attribute を事前フィルタと再ランキングに使う方針が固定されている。
- 既存失敗シナリオの再評価条件が定義されている。

## 9. 引継ぎ時に伝えるべき注意点

- `knowledge_system` は本体実装ではないため、ここで作るのは整理資料と PoC 補助スクリプトまでに留める判断が自然である。
- viewer 側の現状 `tags` は軽量 metadata の一部であり、ここへ寄せすぎると正本がぶれる。
- 2D ICAD は OCR 前提に寄せるのではなく、ネイティブ抽出を先に検証する。
- STEP は viewer の表示変換と検索属性抽出を混同しない。
- RAG の課題は自然言語処理だけでなく、図面管理の属性整備でかなり改善できる前提で考える。
