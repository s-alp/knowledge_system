# Windows側確認事項(抽出・タグ付け改善)

- 作成日: 2026-07-17
- 背景: Cloud検証と抽出内容レビューで、「実ICAD・実図面・SXNET実機でしか確認できない事項」と
  「確認結果が次の実装仕様を決める事項」を切り分けた。本書はWindows側での確認手順をまとめる。
- 同日実装済み(確認不要、反映のみ):
  - 部品数の外部パーツ基準化(内部パーツは参考属性へ)
  - 尺度の `1:6` / `S=1:6` パターン判定(候補1種類のときのみ確定)
  - 熱処理辞書(焼入れ/焼戻し/焼なまし/焼ならし/調質/浸炭/窒化/高周波/真空)と
    硬度指定(HRC/HV/HB等)の抽出・タグ化
- 実装反映手順(Windows実データ):

```powershell
git pull origin master
backend\.venv\Scripts\python.exe -m pytest backend\apps\drawing_metadata\tests
backend\.venv\Scripts\python.exe backend\manage.py re_normalize_snapshots
```

  ※ 部品数は knowledge-entities API が都度計算するため再正規化不要。
  尺度・熱処理・硬度は re_normalize で既存図面の canonical/タグへ反映される。

## A. 幾何公差の「種別」がSXNETから取れるか【優先: 高】

- 現状: `SxGeomTol` / `SxGeomTolDatum` の存在と件数のみ。平行度・位置度・平面度などの記号種別、公差値、データム参照は未取得。
- 確認したいこと:
  1. `SxEntSeg.getGeomList()` が返す `SxGeomTol` オブジェクトに、記号種別(tol type)・公差値・データム名のプロパティがあるか。
  2. あるなら、プロパティ名と値の型(コード値か文字列か)。
- 手順:
  1. 幾何公差(2種類以上)を含む2D図面を1枚選ぶ。
  2. `IcadExtraction.Runner extract --source-kind 2d` を実行し、出力JSONの `geometry_primitives` の `summary` を確認する。
  3. summaryに種別が出ていなければ、sxnet の SxGeomTol 型のプロパティを(オブジェクトブラウザ/リフレクションログで)列挙する。
- 成果物: 種別コード対応表(例: 1=平行度…)とサンプル図面の抽出JSON。
- これが取れると: 「幾何公差:平行度」などのタグ/属性化、検査難易度での検索が可能になる。

## B. 溶接記号の「種別」判別の可否【優先: 高】

- 現状: 溶接注記のテキスト(`weld_notes[].text`)を取得するのみ。すみ肉/開先/現場溶接などの分類なし。
- 確認したいこと: SXNETの溶接記号エンティティ(SxGeomWeld系)に記号種別・脚長・現場溶接フラグのプロパティがあるか。テキストのみの場合、実図面の weld_notes テキストにどんな値が入るか(記号がテキスト化されるのか)。
- 手順: 溶接記号を複数種使った製缶図面で抽出し、`weld_notes` と `geometry_primitives` を確認。
- 成果物: 実図面での weld_notes 値の実例10件と、種別プロパティの有無。

## C. 図枠のラベルと値が「別テキスト要素」になっている実例【優先: 高】

- 現状: 図枠候補は「同一テキスト要素の同じ行(または次の行)」でラベルと値をペア化している。
  ラベルセルと値セルが別のテキスト要素に分かれた図枠では値を拾えない。
  座標は候補に記録済みだが、近傍探索によるペアリングは未実装。
- 確認したいこと:
  1. 主要客先の図枠で、ラベル(「材質」等)と値(「SS400」等)が同一テキストか別テキストか。
  2. 別テキストの場合、ラベルと値の座標距離(X方向・Y方向のオフセット量)の傾向。
- 手順: 代表図枠3種(客先別)の抽出JSONで、図枠領域のテキストの `position_x/y` とラベル・値の並びを確認する。
- 成果物: 「図枠レイアウト実例表」(ラベル/値の分離有無、座標オフセット)。
- これが決まると: 座標ベースのラベル・値ペアリング(近傍探索)を実装できる。表題欄・訂正欄の領域推定(右下・右上)もこの結果を見て設計する。

## D. PRFXの実フィールド名と値形式【優先: 高】

- 現状: 3Dはパーツ付加情報のフィールド名に prfx/prefix/pfx を含む場合(`User_PRFX` 等)のみ、
  2Dは図枠に「PRFX」ラベルがある場合のみ拾う。**「0030」のような値だけの形は拾えない**。
- 確認したいこと:
  1. 実ICADのパーツ付加情報で PRFX に相当するフィールドの実名(客先ごと)。
  2. 図枠上のPRFXは「ラベル付き」か「値のみ」か。値のみの場合の形式(桁数・英字有無。例: CAA5012型 / 0030型)。
  3. ユニット番号(`User_UNIT_NO` 等)も同様に実名と形式。
- 手順: 共有39件+コマツ小山/広島アルミの代表ICDで、抽出JSONの `part_ex_info_fields` と `title_block_candidates` を確認する。
- 成果物: 客先別「PRFX/ユニット番号 形式表」(フィールド名、値パターン、図枠上の位置)。
- これが決まると: 値パターン(正規表現)+客先辞書でのPRFX判定を実装できる。

## E. 熱処理・硬度注記の実例収集【優先: 中】

- 今回実装した辞書とパターン:
  - 熱処理: 焼入れ/焼戻し/焼なまし(焼鈍)/焼ならし(焼準)/調質/浸炭/窒化(タフトライド)/高周波焼入れ/真空焼入れ
  - 硬度: `HRC58`, `HRC58-62`, `HV500`, `HB230` 等(HRC/HRB/HRA/HV/HB/HBW/HS)
- 確認したいこと: 実図面の熱処理注記でこの辞書から漏れる表記(略記・客先固有語)。
- 手順: `re_normalize_snapshots` 実行後、`熱処理:` タグの付与件数と、熱処理があるはずなのに付かない図面の注記原文を収集する。
- 成果物: 取りこぼし表記の一覧(→ `HEAT_TREATMENT_KEYWORDS` へ追記)。

## F. 尺度表記の実態確認【優先: 中】

- 今回実装: トークン全体が `1:6` / `1/6` / `S=1:6` / `尺度 1:6` の形のときに尺度候補とし、
  候補が1種類のときだけ `scale` を確定する。片側が1でない比(7/16等)は除外。
- 確認したいこと:
  1. 実図面の尺度表記の形(S=付きか、裸か、図枠欄か)。
  2. テーパ表記(`テーパ1:10` は除外済みだが、裸の `1:10` テーパ併記)で誤確定しないか。
- 手順: re_normalize 後に `scale` / `scale_candidates` を数枚で確認する。
- 成果物: 誤検出・未検出の実例(あればルール調整)。

## G. 部品数(外部パーツ基準)の妥当性確認【優先: 中】

- 今回実装: 一覧・詳細の部品数(descendantPartCount/childPartCount)を**外部参照パーツのみ**に変更。
  子構造を持つ外部参照はサブアセンブリ候補(childAssemblyCount)。内部パーツ数は参考属性のみ。
- 確認したいこと:
  1. 実アセンブリ(例: 47323200X40c)で外部参照数(28)がBOM感覚と合うか。
  2. ミラー部品・同一部品の複数参照をどう数えるべきか(現状: 出現回数で数える)。
- 手順: knowledge-entities 一覧で部品数を実BOMと突き合わせる。
- 成果物: 数え方の確定(ユニーク数か出現数か、ミラーの扱い)。

## H. 客先・装置・案件辞書とフォルダ命名規則【優先: 高(タグの効き筋)】

- 現状: 辞書はコード内seedのみ(客先3・装置3・メーカー1・規格1)。共有39件の実客先
  (不二越・アイリスオーヤマ等)は拾えず、タグが材質偏重になる主因。
  ファイルパスは検索トークンには入るが、フォルダ階層からの案件名抽出は未実装。
- 確認したいこと:
  1. 対象客先の正式名と略名・通称の一覧(辞書へ投入する語彙)。
  2. フォルダ命名規則の実例(例: `J:\不二越5\260703_次期円筒研削盤開発(竹中様)\作業`)。
     `日付_案件名(担当者様)` パターンで統一されているか、客先でパターンが違うか。
- 成果物: 客先・装置・案件の辞書表と、パス解析ルール仕様(→ project_name 自動抽出の実装へ)。
