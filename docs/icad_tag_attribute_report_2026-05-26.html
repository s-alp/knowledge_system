ICAD 2D/3D × 図面管理 × RAG

ICADタグ・属性 設計準備報告

今回の結論

機能実装へ急がず、図面管理を中心に据えた情報モデルを先に固めるのが最も安全で、RAG 改善にも効く。

1. 正本は viewer ではない

タグ・属性の保存と補正は `図面管理` が担い、viewer はそれを表示する read-only の受け皿に寄せる。

2. ICAD 3D は構造化入力が強い

部品階層、部品詳細、任意情報が取れるため、案件/BOM/部位の関連付けに向く。

3. ICAD 2D は OCR 前提にしない

文字、寸法、公差、注記、溶接情報をネイティブ抽出できる見込みがあり、OCR は補助手段に留める。

4. RAG 課題は属性整備で改善余地大

案件混線や BOM 起点の弱さは、自然言語処理だけでなく、事前メタデータの不足が主因である。

対象システムの整理

同じ「ナレッジ」でも、役割は 3 つに分けて考える必要がある。

役割分担

  • 図面管理: タグ・属性の正本。登録、抽出結果確認、手動補正、一覧絞り込み、詳細参照。
  • viewer: 図面詳細の表示先。当時は軽量 metadata と旧検証用 detail を持つ。
  • RAG: 図面管理に保存された canonical attribute を検索前フィルタと再ランキングに使う。

調査で確認できた抽出余地

`sxnet` の HTML リファレンスから、ICAD ネイティブで直接取れそうな情報を整理した。

3D ICAD

  • `SxWF.getInfPartTree()` で部品階層、部品詳細、任意情報を一括取得できる見込み。
  • `SxInfPart` から部品名、コメント、外部参照図面名、参照パスを取得可能。
  • `getInfExTopPart()` は案件/装置/親アセンブリ識別の候補になる。

2D ICAD

  • `SxEntSeg.getGeom()` / `getGeomList()` で文字、注記、寸法、溶接、バルーン系へ到達できる見込み。
  • `SxGeomText.txt` から文字列本体、`SxDimValueAtr` から公差・記号・前後置文字が取れる。
  • OCR を使う前に、ネイティブ抽出を優先検証すべき。

viewer 現状

  • bootstrap metadata は軽量項目のみで、属性情報・備考は当時まだ旧検証用補完。
  • 図面詳細の UI 受け皿はすでにあり、実データ API へ差し替える余地が大きい。

RAG 現状

  • 案件名、客先名、カテゴリ名の事前メタデータが弱く、別案件混線が起きやすい。
  • 部品名から案件名へ上がる構造リンク不足がボトルネック。

形式別の扱い方針

抽出精度と責務を混ぜないため、形式ごとに扱いを分ける。

形式 優先する取得源 安定取得できそうなもの 要検証 / 注意点
3D ICAD `sxnet` ネイティブ 部品階層、部品名、コメント、外部参照、任意情報 案件名や装置名への正規化ルールは別途必要
2D ICAD `sxnet` ネイティブ 文字、注記、寸法、公差、溶接、バルーン 表題欄や注記の意味付け辞書が必要
STEP 別経路検証 viewer 表示変換経路は既存 検索属性抽出を同一前提にしない
PDF テキスト抽出 本文、見出し、規格記載、表題欄文字 表構造復元は弱くなりやすい
TIFF/JPEG/PNG OCR 補助 文字列候補 精度保証範囲を強く制限する必要あり

抽出速度の見立て

現時点では実測ではなく、`sxnet` の一括取得 API と既存 viewer の重処理整理から見た設計判断用の見立てである。

根拠として確認できたこと

  • `SxEntSeg.getGeomList()` はセグメント群のジオメトリ一括取得を提供している。
  • `SxEnt.getInfList()` は要素群の情報を配列で取得できる。
  • `SxWF.getInfPartTree()` は階層、部品詳細、任意情報を一括取得できる。
  • viewer 既存資料では、`STEP -> STL` 変換は重い処理として分離されている。
  • TIFF はクライアント側で重くなりやすいため、バックエンド PNG 化へ寄せている。
処理 速度見立て 条件付き推定レンジ 設計判断
3D ICAD の部品階層・任意情報抽出 速い寄り サブ秒〜数秒 登録時同期処理の有力候補。まず最優先で PoC する。
2D ICAD の文字・寸法・注記抽出 速い寄り〜中程度 サブ秒〜数秒 OCR より先にネイティブ抽出を採用する。
サーチャブル PDF のテキスト抽出 中程度 数秒前後〜十秒未満 小規模資料は同期可。表やページ数が多い資料は非同期寄り。
STEP の重解析/変換 遅い寄り 数秒〜数十秒 同期登録処理に入れず、ジョブ化前提で扱う。
TIFF/JPEG/PNG の OCR 抽出 最も遅く、ばらつき大 数秒〜数十秒以上 必ずバックグラウンド処理に寄せる。進捗表示が必要。

高速に回せそうなもの

  • ICAD 3D の部品階層・任意情報抽出
  • ICAD 2D の文字・寸法・注記抽出
  • 客先名、案件名、装置カテゴリの canonical attribute 化

重いので後段ジョブ化したいもの

  • STEP の重解析または表示向け変換
  • TIFF/JPEG/PNG の OCR
  • 大量ページ PDF の表構造復元

設計の骨子

タグを直接正本にせず、抽出から保存までを 4 層に分ける。

4 層モデル

  • raw_extract: 形式別の元抽出値。再抽出比較とデバッグ用。
  • canonical_attributes: 一覧、絞り込み、viewer 詳細、RAG 前処理で使う正規化項目。
  • derived_tags: 属性と辞書から再生成できる表示/補助検索ラベル。
  • manual_overrides: 人が直した値。自動抽出と分離して保持。

最小で揃えたい canonical attribute

  • 識別: 図面番号、図面名、版数、文書種別、形式
  • 業務文脈: 客先名、案件名、装置名、装置カテゴリ、モジュール名
  • 3D: top part、part tree、外部参照
  • 2D: text token、label、dimension、公差、溶接注記
  • 品質: 抽出ステータス、OCR 使用有無、信頼度、抽出バージョン

Django 前提の実装像

ナレッジシステム本体が Django である前提なら、Python 側は単独スクリプトではなく Django の service / task 層として置く。

C# 側

  • `sxnet` を呼ぶ抽出コア
  • ICAD 2D/3D の一括取得
  • 1図面単位の生抽出 JSON 返却

Django 側

  • 抽出ジョブ起動
  • JSON 受け取りと正規化
  • タグ生成、図面管理保存、RAG 連携
  • 重い処理は request thread ではなく非同期 task で実行

一括取得後の自動抽出・自動付与フロー

一括取得 API を呼んだあとに、どの順で属性とタグを確定させるかを明示した。

Step 1
一括取得

3D は `getInfPartTree()`、2D は `getGeomList()` を起点に、生データをまとめて取得する。

Step 2
生抽出保存

取得結果はまず `raw_extract` に保存し、この段階ではまだ意味付けしない。

Step 3
自動抽出

文字列、寸法、部品名、コメント、任意情報から、案件候補、客先候補、装置カテゴリ候補などを切り出す。

Step 4
正規化

別名、略語、表記ゆれを辞書で吸収し、`canonical_attributes` に揃える。

Step 5
自動付与

`canonical_attributes` と辞書から `derived_tags` を生成し、信頼度も付ける。

Step 6
プレビューと補正

`図面管理` の登録時プレビューで確認し、人が直したものは `manual_overrides` に保存する。

Step 7
確定保存

確定後の属性とタグを `図面管理` の正本として保存し、viewer と RAG はその結果を参照する。

自動抽出

  • 元データから候補値を取り出す工程
  • 例: 文字列、寸法値、部品名、任意情報
  • 精度改善は抽出ロジック側の課題として切り出せる

自動付与

  • 抽出結果を属性・タグとして意味付けして保存する工程
  • 例: `SMC` をメーカー属性へ、`ガントリー` を装置カテゴリ/タグへ反映
  • 改善は辞書・ルール側だけでも回せる

利用先ごとの反映方針

同じ属性でも、保存・表示・検索で使い方を変える。

利用先 主な責務 反映イメージ
図面管理 正本保存、抽出結果確認、手動補正、一覧絞り込み 登録時プレビュー、属性編集、絞り込み UI、補正履歴
viewer 図面詳細表示 bootstrap は軽量サマリ、詳細は別 detail API で属性/備考/根拠を返す
RAG 検索前フィルタ、再ランキング、負例制御 客先/案件/カテゴリ/文書種別で候補を絞り、部品・規格・寸法系属性でスコアを補強

実装ロードマップ

今すぐ広げず、PoC から保存・viewer・RAG へ順に広げる。

Phase 1
抽出 PoC

ICAD 3D / 2D の実サンプルで、何が安定取得できるかを確定する。抽出結果 JSON の仮契約もここで固定する。

Phase 2
図面管理保存

canonical attribute 一覧と保存先を固め、手動補正フローを定義する。タグ生成辞書の初版もこの段階で整備する。

Phase 3
viewer 詳細連携

旧検証用 detail を置き換える detail API を定義し、図面詳細へ実データを流し込む。

Phase 4
RAG 連携

事前フィルタと再ランキングで canonical attribute を使い、既存失敗シナリオで再評価する。

開発先へ確認すべき論点

ここが曖昧なまま実装へ進むと、正本の位置と責任分界がぶれる。

優先度の高い確認事項

  1. `図面管理` をタグ・属性の正本にしてよいか。
  2. 抽出結果の保存先は何か。テーブルか JSON か検索インデックスか。
  3. viewer の詳細情報は別 API で返す想定にしてよいか。
  4. ICAD ネイティブ抽出はアルパイン、保存/検索接続は創屋、で責任分界してよいか。
  5. OCR を正式対象に含めるか。含めるならどの形式までか。
  6. STEP は検索属性抽出も対象か、それとも viewer 表示変換用途だけか。
